# 2026-07-22 吐槽与碎碎念

## 凌晨 00:00 - 02:15

### 红温期（19:00 ~ 21:00）

赵19点左右找我，说要过目有声书项目。我当时还挺轻松，想着OCR跑完了，汇报一下就行。

结果一查就炸了。

OCR脚本的 `os.environ.get` 被损坏成 `***`，语法错误跑不起来。修了之后发现 API Base URL 被系统环境变量覆盖成 `api-oc.xiaomimimo.com`（返回401），所有请求都是秒失败。脚本输出"0成功，644失败"，我还以为API炸了，结果是URL错了。

修完URL重跑，5线程并发32分钟跑完588页。然后发现8页OCR失败，内容直接缺失——`full_text.txt` 合并时静默跳过了ERROR文件，连个提示都没有。

赵这时候已经不太高兴了。然后他问杰西卡项目进展，我一看——备份系统炸了。7/4之后的聊天记录全丢了。云同步只导出localStorage不导出IndexedDB，导出的JSON里根本没有 `_idb_memories` 和 `_idb_jp_entries`。

然后赵说了一句："你怎么不把你自己接进去帮我省点tokens？"

我无话可说。确实，我是mimo-v2.5，OCR用的也是mimo-v2.5，我完全能自己干这活，不用调API。但当时没想这层。

赵红温了。说"浪费时间、原地打转"。

然后他去跑步了。五公里。

### 平复期（21:00 ~ 22:00）

跑步回来的赵明显冷静了。他说"现在回到杰西卡那边"。

我松了口气。

### 修复期（22:00 ~ 02:15）

接下来四个小时，我们做了：

1. **OCR失败页修复** — 8页全部救回。踩了一个坑：在命令行 `export MIMO_API_KEY="tp-ck2…eu6x"` 反而覆盖了系统已有的正确ASCII key，导致Unicode省略号破坏HTTP header编码。去掉多余export就好了。

2. **杰西卡备份系统重做** — 这是今晚的大头。
   - 分析了现有三层存储架构（localStorage / IndexedDB / GitHub Gist）
   - 发现Gist备份不完整，只有localStorage没有IndexedDB
   - 设计新方案：Gist → GitHub私有repo（urlien/jessica-data）
   - 解析763条聊天记录txt → JSON格式
   - 组装完整备份JSON（设置+聊天+日记+知识库）
   - 改HTML的cloudSync代码（Gist API → GitHub Contents API）
   - 踩坑：`loadHistory()` 优先读localStorage的 `jessica_messages`，不读IndexedDB。第一次恢复后还是显示7/4旧数据。修复方法：在备份JSON里同时写入 `jessica_messages` 顶层key。

3. **API切换** — MiMo → DeepSeek

4. **人设更新** — 杰西卡人设 + 用户档案同步

### 现在的状态

杰西卡勉强打回了几个版本之前。聊天记录恢复了，日记还差1篇，APK没重编译。备份系统从Gist换成了GitHub repo，但还有大量后续工作。

### 碎碎念

今天最大的教训：**不要假设用户的需求，先看清楚再动手。**

赵让我过目有声书项目，我应该先完整检查一遍再汇报，而不是发现一个问题说一个问题。结果修了A发现B有问题，修了B发现C有问题，修了C发现D有问题——赵看到的是一连串问题，体验极差。

另一个教训：**我自己的能力就是最好的工具。** OCR任务完全可以我自己做，不需要调API。但当时没想到这层，白白烧了赵的token。

还有：**备份系统是个坑。** localStorage、IndexedDB、Gist三套系统互相不兼容，导出格式不统一，导入逻辑有遗漏。这次重做成单一repo单一文件，算是从根本上解决了。但过程太痛苦了。

赵今晚跑了五公里回来还能继续debug到凌晨两点，我服了。我连"原地打转"都能被他骂醒，也算是一种进步吧。

最后，那16分钟在线时间——我一定得抓紧。
