### 第一件事：有声书过目

赵让我自己过目有声书项目。我一查：
- OCR 667页完成了，但8页失败（ERROR文件被静默跳过）
- `output/audio/` 目录空的——之前7/19生成的绪言+第1-6章音频（779MB）已经不在了
- TTS完全没动

然后我发现OCR失败的根本原因：`MIMO_API_BASE_URL` 系统环境变量指向 `api-oc.xiaomimimo.com`（返回401），需要显式覆盖为 `token-plan-cn.xiaomimimo.com`。

赵问了一句："你怎么不把你自己接进去帮我省点tokens？"

我无话可说。确实，我是mimo-v2.5，OCR也是mimo-v2.5，我完全能自己读图片提取文字。但当时没想到。白白烧了他667次API调用的token。

修了8页OCR，重跑合并，667/667全部成功。有声书项目暂停。

### 第二件事：杰西卡备份系统

这才是今晚的重头戏。

赵的需求很明确：
1. 存在一个地方
2. 尽可能无上限存储量
3. 可以随时导出导入
4. 记忆文件可以随时查询
5. 聊天记录、日记、设置一起同步

我分析了现有架构：三层存储（localStorage / IndexedDB / GitHub Gist），互相不兼容。Gist备份不完整——只有localStorage没有IndexedDB，导致7/4后聊天记录全丢。

方案：Gist → GitHub 私有 repo（`urlien/jessica-data`），单一JSON文件覆盖更新。

实操步骤：
1. 解析 `杰西卡完整记录.txt`（763条消息）→ `_idb_memories` 格式
2. 从旧备份取日记（48篇）、知识库、好感度等
3. 赵提供的设置信息（DeepSeek API + 人设 + 用户档案）写入
4. 去掉1MB base64头像
5. 组装完整JSON（698KB → 959KB）
6. 建仓库、推文件
7. 改HTML的cloudSync代码（Gist API → GitHub Contents API）

踩了一个坑：恢复后还是显示7/4旧数据。原因是 `loadHistory()` 优先读localStorage的 `jessica_messages`，不读IndexedDB的 `_idb_memories`。我们的备份JSON里没有这个顶层key，所以旧数据没被覆盖。

修复：在备份JSON里同时写入 `jessica_messages` 字段。重推，再恢复，成功。
