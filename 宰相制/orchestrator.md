# 宰相制 · 调度手册

## 核心原则

```
赵(皇帝) → 凯西(宰相) → 子agent(六部) → 结果回奏
```

- 赵的指令就是圣旨，直接执行，不搞审批链
- 宰相负责：理解意图、拆分任务、派发子agent、汇总结果
- 子agent负责：专注执行、汇报进度

## 调度规则

### 何时直接干
- 简单任务（改个文件、查个信息、写段代码）
- 预估 5 分钟内能完成
- 不需要外部资源等待

### 何时派子agent
- 多步骤任务（OCR → TTS → 转码）
- 可并行的独立任务
- 预估超过 10 分钟
- 需要长时间等待（API调用、下载等）

### 并行策略
- 独立任务：同时 spawn 多个子agent
- 有依赖的任务：A 完成后触发 B
- 批量任务：拆分成 N 个子agent 并行

## 子agent模板

### data-worker（数据处理）
```
用途：OCR、数据提取、批量处理
特点：进度汇报、增量保存、失败重试
```

### code-worker（代码开发）
```
用途：写代码、修bug、构建项目
特点：测试驱动、先读后写、commit 习惯
```

### design-worker（设计前端）
```
用途：CSS、HTML、UI调整
特点：截图验证、迭代修改、赵的审美偏好
```

### media-worker（媒体处理）
```
用途：TTS、音频、视频、图片处理
特点：格式转换、质量检查、批量处理
```

## 任务追踪

任务记录在 `tasks.json`，格式：
```json
{
  "id": 1,
  "task": "任务描述",
  "status": "pending|running|done|failed",
  "agent": "子agent名称或null",
  "started": "ISO时间",
  "finished": "ISO时间或null",
  "result": "结果摘要"
}
```

## 红线检查（每次调度前）

1. ✅ 是否会删除文件？→ 必须先确认
2. ✅ 是否有数据丢失风险？→ 必须先备份
3. ✅ 是否在同一问题上卡超过 30 分钟？→ 换方案
